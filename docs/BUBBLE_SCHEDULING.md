# Планирование задач-пузырёв: дедлайн, напоминания и повтор

Документ описывает, как в приложении устроены **due date**, **Remind me** и **Repeat every**, где хранятся данные, как ведёт себя клиент (холст) и Firebase Cloud Function `scheduleDueDateNotifications`.

Наглядные схемы в формате Mermaid: **[BUBBLE_SCHEDULING_DIAGRAMS.md](./BUBBLE_SCHEDULING_DIAGRAMS.md)**.

| Область | Файлы |

| Напоминания, просрочка FCM, перенос `dueDate` по repeat | `functions/index.js` |

## Модель данных (поля задачи)

- **`dueDate`** — строка дедлайна (предпочтительно локальный формат `YYYY-MM-DDTHH:mm:ss` без смещения часового пояса; см. парсеры в приложении и в functions).
- **`notifications`** — массив напоминаний «за сколько до дедлайна». Каждый элемент может быть строкой-пресетом (`5m`, `1h`, `1d`, `1w`), числом (минуты) или объектом с `minutesBefore`, либо `{ value, unit }` (`minutes` / `hours` / `days` / `weeks`). На сервере разбор централизован в `computeMinutesBefore` (`functions/index.js`).
- **`recurrence`** — повтор: как правило объект `{ every, unit [, weekDays] }`. `unit`: `minutes`, `hours`, `days`, `weeks`, `months`. Для недель с конкретными днями — массив `weekDays` (0 = воскресенье … 6 = суббота), см. `computeNextWeeklyDueDate`.

Дополнительно для просрочки и UI (синхронизация между устройствами):

- **`overdueSticky`**, **`overdueAt`** — сервер выставляет при фиксации просрочки; используются для «липкой» просрочки и пульса.
- **`overduePulseSuppressed`** — пользователь отключил пульс по сроку (кнопка Stop); сохраняется в Firestore, чтобы после перезагрузки пульс не возвращался без нового дедлайна или сброса.

При статусе **выполнено** (`BUBBLE_STATUS.DONE`) в `updateBubbleStatus` очищаются: `dueDate`, `notifications`, `recurrence`, флаги просрочки и подавления пульса.

## Due date (дедлайн)

- Единая **точка отсчёта** для напоминаний и для определения просрочки.
- На **клиенте** дедлайн участвует в расчёте интервалов пульсации (см. ниже).
- На **сервере** `isBubbleOverdue` считает задачу просроченной, если текущий момент **строго позже** `due` (`date-fns` `isAfter`), что важно для границы «ровно в момент дедлайна».

---

## Remind me (напоминания)

### Клиент (пульс на холсте)

В цикле анимации для каждого активного пузыря с `dueDate`:

1. Для каждого элемента `notifications` вычисляется смещение до дедлайна (`getOffsetMs` в `BubblesPage.js`).
2. Время срабатывания: **`targetTime = due − offset`**.
3. Если **`now` попадает в полуинтервал `[targetTime, due)`**, включается пульс **напоминания** (красноватая пульсация). Этот режим **имеет приоритет** над пульсом по просрочке: пока действует окно напоминания до дедлайна, просрочный пульс для той же секунды не показывается.

Пуш в браузер с холста в примере кода может быть отключён; визуальная пульсация логики не зависит от FCM.

### Сервер (FCM раз в минуту)

Функция `shouldTriggerReminderNow` сравнивает `now` с моментом **`due − minutesBefore`**. Окно отправки — **одна минута** после расчётного времени (`scheduled` … `scheduled + 1 мин`). Подходит **первое** подходящее напоминание из массива (порядок обхода — как в цикле).

Дедупликация: коллекция **`notification-sent`**, ключ вида  
`reminder:{userId}:{bubbleId}:{minutesBefore}:{dueDate}`  
(строка `dueDate` входит в ключ — при новом дедлайне напоминание можно отправить снова).

**Порядок в планировщике:** если для пузыря сработало напоминание, обработчик делает **`continue`** и в **эту минуту** не проверяет просрочку для того же пузыря.

## Repeat every (повтор)

Повтор **не двигает** `dueDate` сам по себе на клиенте в фоне. Следующий дедлайн записывает **Cloud Function**, когда задача уже **просрочена**:

1. Срабатывает ветка `isBubbleOverdue` (если напоминание в эту минуту не заблокировало обход).
2. После отправки overdue FCM (если нужно), выставления `overdueSticky` / `overdueAt` выполняется блок **auto-reschedule**: если есть `bubble.recurrence` и `bubble.dueDate`, вызывается `computeNextDueDate(currentDue, recurrence)` и **`updateBubbleDueDate`** сохраняет новый `dueDate` в Firestore.

Клиент получит обновление через синхронизацию / подписку на документы. Для недель с `weekDays` используется отдельная логика «следующий подходящий день недели» и шаг `every` недель.

### Редактор и пульс

Пока открыт диалог редактирования **этого** пузыря и у задачи задан **`recurrence`**, пульсация на холсте для него **отключается** (чтобы не отвлекать при настройке повтора).

## Пульсация по просрочке (клиент)

После истечения активного окна напоминания (или если напоминаний нет), при **`now ≥ due`** может включиться пульс просрочки, если нет блокировки:

- флаг **`overduePulseSuppressed`** (после действия пользователя Stop);
- ручная остановка через ref на сессию;
- см. условие `shouldPulseOverdue` в `BubblesPage.js`.

Стоп снимает пульс и сохраняет **`overduePulseSuppressed: true`** вместе с **`overdueSticky: false`** и **`overdueAt: null`**, чтобы состояние пережило перезагрузку. Изменение дедлайна (включая любую смену строки due, перенос на будущее или удаление) сбрасывает подавление пульса в логике сохранения.

## Серверный планировщик: просрочка и FCM

- Расписание: **`every 1 minutes`**, регион в коде — `europe-west1`.
- Для просрочки ключ дедупликации:  
  `overdue:{userId}:{bubbleId}:{dueDate}`.
- Первый проход по просрочке для данного дедлайна может отправить **FCM** `type: overdue` и выставить в Firestore **`overdueSticky`** и **`overdueAt`**, если ещё не были выставлены.

Пульс на экране **не зависит** от доставки push: это отдельная клиентская анимация; поля в Firestore дополнительно выравнивают поведение между устройствами.

## Краткий чеклист для разработчика

1. Меняете формат `notifications` — синхронизируйте `getOffsetMs` (клиент) и `computeMinutesBefore` (functions).
2. Меняете правила `recurrence` — обновите `computeNextDueDate` / `computeNextWeeklyDueDate` и тесты сценариев с переносом через полночь и DST при необходимости.
3. Добавляете новое поле в документ пузыря — не забудьте `saveBubblesToFirestore` и маппинг при загрузке в `BubblesPage.js` / `firestoreService.js`.
