# Code review — 2026-06-10

**TL;DR:** архитектура безопасности в целом здоровая (правила Firestore owner-only с default deny, весь HTML проходит через DOMPurify, продуманная работа с таймзонами в Cloud Functions). Но есть три серьёзные проблемы: **полная перезапись всех задач в Firestore при каждом изменении** (главный источник затрат и лишних вызовов функций), **нарушение Rules of Hooks в HtmlRenderer** (потенциальный крэш) и **`.env.production` с `public/sw.js`, отслеживаемые в git** вопреки `.gitignore`. Ниже — всё по категориям.

## 🔴 Критичное

### 1. `.env.production` и `public/sw.js` закоммичены в git
`git ls-files` показывает оба файла как отслеживаемые, хотя оба перечислены в `.gitignore` (gitignore не действует на уже добавленные файлы). CI генерирует их из секретов (`deploy.yml:57`), то есть в репозитории лежит устаревшая копия с реальной Firebase-конфигурацией и VAPID-ключом. Web-ключи Firebase публичны по дизайну, так что утечки секретов как таковой нет, но это нарушает вашу же политику из CLAUDE.md и создаёт привычку коммитить env-файлы. Фикс: `git rm --cached .env.production public/sw.js` (история останется — при желании вычистить, но необходимости нет).

### 2. Нарушение Rules of Hooks в `HtmlRenderer.js:17-26`
`useMemo` вызывается **после** условного `return null`. Если у смонтированного компонента `html` сменится с непустого на пустой (пользователь стёр описание) — React бросит «Rendered fewer hooks than expected» и приложение упадёт в белый экран (error boundary нет). Лечится переносом early return ниже `useMemo`. Связанное: в проекте **вообще нет конфигурации ESLint** (ни `eslintConfig` в package.json, ни `.eslintrc`) — поэтому `react-hooks/rules-of-hooks` это не поймал.

### 3. Полная перезапись всех задач при каждом изменении — `firestoreService.js:69-122`
`saveBubblesToFirestore` на каждый вызов: читает **все** документы (`getDocs`, строка 76), затем batch-пишет **все** задачи через `set merge`. Вызывается при создании задачи, сохранении правки, удалении тега, переключении rich-text, остановке пульсации, импорте и стартовом cleanup. Последствия при N задач на одно действие:
- N reads + N writes в Firestore;
- N срабатываний триггера `maintainNextNotifyAt` (каждая запись будит функцию, даже если данные не изменились — гард `significantChanged` лишь быстро выходит, но invocation и запись уже оплачены);
- **`writeBatch` ограничен 500 операциями** — при ~499+ задачах `commit()` упадёт целиком, и код свалится в legacy-fallback (строка 105), который запишет весь массив в родительский документ (лимит 1 MiB) — путь к потере данных.

Правильный паттерн уже есть в этом же файле: `updateBubbleStatus` (строка 188) пишет один документ через `updateDoc`. Создание/правку стоит перевести на такие же точечные операции.

## 🟠 Баги

**4. Side effects внутри `setState`-апдейтеров.** `saveBubblesToFirestore(updated)` вызывается прямо внутри колбэка `setBubbles(prev => ...)`: `BubblesPage.js:884`, `980`, `1241`, `1923`, `2625`. Апдейтеры должны быть чистыми; приложение обёрнуто в `<React.StrictMode>` (`index.js:25`), где апдейтеры в dev вызываются дважды → **двойные batch-записи и двойные срабатывания триггеров** на каждое действие в dev. Сохранение нужно выносить за пределы апдейтера.

**5. Гонка в `handleDeleteTag` — `BubblesPage.js:1219-1251`.** Таймер на 7 секунд захватывает `tags` замыканием. Если за эти 7 секунд теги изменились (другая вкладка, второй delete), `tags.filter(...)` сработает по устаревшему списку и перезатрёт чужие изменения при `saveTagsToFirestore`.

**6. Гонка в `handleMarkAsDone` — `BubblesPage.js:1096`.** `markBubbleAsDone(selectedBubble.id, bubbles)` вызывается из колбэка анимации (через ~250 мс) со старым `bubbles`; вернувшийся массив целиком кладётся в `setBubbles(updatedBubbles)`, перетирая любые изменения, прилетевшие за время анимации (например, snapshot с сервера).

**7. Мёртвая ветка «live-обновления открытого диалога» — `useMatterEngine.js:204-221`.** Эффект выполняется один раз на маунте, поэтому `editDialog` и `selectedBubble` навсегда зафиксированы как `false`/`null` — блок синхронизации открытого редактора с серверными обновлениями не выполняется никогда. Комментарий в коде это признаёт («stale closure — matches original»), но фактически фича сломана: при автопереносе повторяющейся задачи открытый диалог не обновится.

**8. «Призрачные» тела при удалении с другого устройства — `useMatterEngine.js:171`.** Когда snapshot приносит список без задачи, она исчезает из state, но её Matter-тело никто не удаляет из `engine.world` (эффект в `BubblesPage.js:664` итерирует только текущие `bubbles`). На канвасе остаётся пустой кружок без текста до перезагрузки.

**9. Возможные дубли уведомлений в Cloud Functions — `functions/index.js:249-258, 518`.** `wasNotificationSent`/`markNotificationSent` — это check-then-set без транзакции, а `onSchedule` каждую минуту с `maxInstances: 10` допускает наложение прогонов (если прогон длится >1 мин). Два параллельных прогона могут отправить одно и то же уведомление дважды. Фикс — `create()` вместо `set()` (упадёт, если ключ уже есть) и проверка до отправки.

**10. Cleanup на `getMinutes() === 0` — `functions/index.js:525`.** Schedule `every 1 minutes` дрейфует по секундам; для интервала в 1 минуту риск проскочить нулевую минуту мал, но надёжнее отдельный cron `0 * * * *`.

**11. Захардкоженный базовый путь.** `/to-round-react/pop.mp3` (`BubblesPage.js:1023`), `/to-round-react/bubbles.png` (`:2089`), `/to-round-react/sw.js` (`index.js:32`) — в локальной разработке (`npm start` на `/`) звук и иконка 404, SW не регистрируется. Правильно — `process.env.PUBLIC_URL`.

**12. Мелкие.** `getOffsetMs` (`BubblesPage.js:2032`) не понимает строковый пресет недель `'Nw'` (вернёт 0), в отличие от серверного парсера `[mhdw]`; на строке для не-`m/h/d` дальше идёт обращение `notification.type` к строке — просто вернёт 0. Удаление FCM-токена по коду `invalid-argument` (`functions/index.js:109`) может снести валидный токен, если невалиден payload, а не токен.

## 🟡 Производительность

**13. Состояние формы живёт на уровне страницы — `BubblesPage.js:249-250`.** `title`/`description` редактируемого диалога — это state 2900-строчного компонента: **каждое нажатие клавиши ре-рендерит всю страницу** (включая списки, панели, вычисление `showStopPulsing` и `getCategoryBubbleCounts()` в рендере). Это главный кандидат на вынос в `EditBubbleDialog`/`CreateBubbleDialog` с локальным состоянием.

**14. `TextOverlay` — компонент, создаваемый через `useCallback` внутри рендера (`BubblesPage.js:1557-1683`).** При каждом изменении зависимостей (`bubbles`, `foundBubblesIds`…) создаётся **новый тип компонента** → React размонтирует и заново монтирует всё поддерево, сбрасывая внутренний state и rAF-цикл. Плюс `setPositions` до 60 раз/сек, пока пузыри движутся — React-рендер оверлея на каждый кадр. Стоит вынести в отдельный мемоизированный компонент.

**15. Pulse-эффект перезапускается на каждом рендере — `BubblesPage.js:1866`.** В зависимостях `getBubbleFillStyle` — обычная функция, пересоздаваемая каждый рендер, поэтому эффект (cancel + restart rAF) дёргается постоянно. Внутри кадра — `tags.find` для каждого пузыря (O(n·m) на 60 fps) и `parseLocalDateTime` каждой задачи каждый кадр. Map тегов по id + `useCallback` для `getBubbleFillStyle` заметно удешевят.

**16. Дублированные/квадратичные эффекты фильтрации.** Эффект `BubblesPage.js:626-643` — подмножество эффекта `:664-767` (оба добавляют тела в мир), первый можно удалить. Внутри обоих `engineRef.current.world.bodies.includes(bubble.body)` — O(world) внутри цикла по задачам → O(n²) на каждое изменение.

**17. Нет code splitting.** Matter.js, TipTap (+ все расширения, включая таблицы), `@xyflow/react`, `mind-elixir`, MUI — всё в одном бандле. `MindMapPage` — очевидный кандидат на `React.lazy` (mind-elixir + xyflow нужны только там); это легко срежет существенную часть initial load.

**18. Двойная загрузка при старте.** `loadBubblesFromFirestore` (read всех задач) + сразу `subscribeToBubblesUpdates` (первый snapshot тоже отдаёт все задачи) — `useMatterEngine.js:95,168`. Можно строить начальное состояние из первого snapshot и убрать отдельный load. Плюс стартовый `cleanupOldDeletedBubbles` при срабатывании делает полную перезапись (см. п. 3).

**19. Поиск по `description` ищет в сыром HTML (`useSearch.js:36`, `TaskList.js:249`).** Запрос «strong» или «span» найдёт все rich-text задачи. Для rich-text стоит искать по тексту без тегов.

## 🟢 Безопасность — заметки и харднинг

- **XSS:** путь отображения закрыт хорошо — единственный `dangerouslySetInnerHTML` в проекте находится в `HtmlRenderer` с whitelist DOMPurify, hook добавляет `rel="noopener noreferrer"`; TipTap Link с `openOnClick: false`; `escapeValue: false` в i18n безопасен, т.к. переводы не попадают в HTML-sinks. Замечание: разрешённый `img src` на любой http(s)-URL — это внешние запросы при просмотре задачи (tracking pixel/утечка IP); в mind-elixir `engineData` рендерится библиотекой как есть — но записать его может только владелец, так что это максимум self-XSS.
- **FCM-токен не удаляется при logout (`authService.js:58-72`).** Документ токена остаётся под старым uid; если на этом же браузере залогинится другой аккаунт, устройство будет получать уведомления (с заголовками задач) **обоих** пользователей. Стоит вызывать `deleteToken(messaging)` и удалять документ токена при выходе.
- **Правила Firestore** корректные (owner-only, default deny, `notification-sent` закрыт), но без валидации данных: владелец может писать документы произвольной формы/размера. Для одного себя это не опасно, но валидация полей в rules защитила бы серверную логику от мусора.
- **`"latest"` в зависимостях** (`firebase`, `@mui/*`, `i18next`, `react-i18next`, `react: ">=16.8.0"`) — пока спасает lockfile с `npm ci`, но любой `npm install` может молча притащить новый major (supply-chain и стабильность). Стоит зафиксировать версии. `react-scripts` 5 тянет известные dev-time уязвимости транзитивно — на прод-бандл не влияют.
- CI-workflow в порядке: `pull_request` (не `pull_request_target`), деплой и пуш версии гейтятся по `main`.
