# Дизайн: снижение стоимости scheduled-уведомлений

- **Дата:** 2026-06-08
- **Статус:** на ревью
- **Затрагивает:** `functions/index.js`, `firestore.indexes.json`, разовый backfill-скрипт
- **Связано:** [cron-тайминг](../../notifications.md#тайминг-расписания-every-n-vs-n)

## Проблема

Cloud Function `scheduleDueDateNotifications` стоит на `every 1 minutes` и каждый прогон
читает **все** активные задачи всех пользователей:

```js
db.collectionGroup('bubbles').where('status', '==', 'active').get()
```

плюс `getUserFcmTokens(userId)` для каждого пользователя — даже когда отправлять нечего.

Главный драйвер затрат на плане Blaze — **Firestore reads**:

> reads ≈ (число активных задач + токенов) × 1440 прогонов в сутки

При тысячах задач это десятки–сотни миллионов чтений в месяц, растёт линейно с числом
задач и тратится вхолостую (99% задач в конкретную минуту уведомление не нужно).

## Ограничения (со слов владельца)

- **Точность:** до 1 минуты. Снижать частоту запуска нельзя → расписание остаётся минутным.
- **Масштаб:** тысячи задач и больше → нужно решение, стоимость которого **не растёт** с числом задач.
- **Legacy-схема мертва:** старый формат (массив `bubbles` в документе `user-bubbles/{uid}`)
  больше не используется. Fallback на него убираем.

## Критерии успеха

1. Reads за прогон ≈ числу фактически отправляемых уведомлений, **а не** числу задач в базе.
2. Уведомления приходят с той же минутной точностью (или лучше — пропуск минуты не теряет напоминание).
3. Cleanup `notification-sent` продолжает работать.
4. Триггер пересчёта поля не зацикливается (нет рекурсии запись→триггер→запись).

## Рассмотренные варианты

- **A. Денормализованное поле `nextNotifyAt` + индексный запрос** — выбран. Стоимость
  определяется числом срабатываний, а не объёмом базы; минутная точность сохраняется;
  ложится на существующую архитектуру (`collectionGroup`, `parseLocalDateTime`).
- **B. Событийная схема на Cloud Tasks** — потолок по дешевизне, но высокая сложность
  (создание/отмена/перепланирование тасков на каждое изменение, по таску на reminder,
  отдельная логика recurrence). Избыточно для текущего масштаба; держим как запасной.
- **C. Только мелкие правки** (токены по условию, `maxInstances`) — не лечит главный
  драйвер (чтение всех задач). Недостаточно.

## Решение A

### Поле `nextNotifyAt`

В каждый документ задачи (`user-bubbles/{uid}/bubbles/{bubbleId}`) добавляется
`nextNotifyAt: Timestamp` — абсолютный момент (UTC) ближайшего **необработанного** события
по задаче. Хранится как Firestore `Timestamp` (нативно индексируется и сравнивается).

Семантика по статусу задачи:
- активная, есть будущие события → ближайший будущий триггерный момент;
- активная, событий в будущем нет, но задача просрочена / recurrence → `now`
  (пусть scheduled-функция обработает немедленно и сдвинет поле);
- неактивная / без `dueDate` → поле отсутствует (range-фильтр `<=` такие документы не вернёт).

### Алгоритм `computeNextNotifyAt(bubble, fromTime)`

«Триггерные моменты» задачи:
- для каждого `notif` из `bubble.notifications`: `reminderTime = dueDate − minutesBefore`
  (`minutesBefore` через существующий `computeMinutesBefore`);
- `overdueTime = dueDate` (момент, с которого шлём overdue).

`dueDate` переводится в абсолютный момент через существующий `parseLocalDateTime(dueDate, tz)`.

Возвращает **минимальный триггерный момент, строго больший `fromTime`**. Если таких нет:
- задача активна и (просрочена или есть `recurrence`) → вернуть `fromTime` (немедленная обработка);
- иначе → `null` (поле не пишем / удаляем).

Функция экспортируется в `exports._test` рядом с остальными (есть паттерн в `functions/test-tz.js`).

### Триггер `onDocumentWritten`

Новый триггер на `user-bubbles/{uid}/bubbles/{bubbleId}`:

1. На удаление документа — выходим.
2. **Гард от рекурсии:** если значимые поля (`dueDate`, `notifications`, `status`,
   `recurrence`) в `before` и `after` совпадают — выходим (значит, запись пришла от нас
   самих, когда мы дописывали `nextNotifyAt`/служебные поля).
3. Иначе считаем `computeNextNotifyAt(after, now)` и пишем `nextNotifyAt`
   (`merge: true`). Если результат `null` — удаляем поле (`FieldValue.delete()`).

### Изменения в scheduled-функции

Заменяем `fetchAllUserBubbles()` на запрос только сработавших задач:

```js
const due = await db.collectionGroup('bubbles')
  .where('status', '==', 'active')
  .where('nextNotifyAt', '<=', admin.firestore.Timestamp.fromDate(now))
  .orderBy('nextNotifyAt')
  .get();
```

Группируем результат по `userId` (как сейчас в `fetchAllUserBubbles`). `getUserFcmTokens`
вызывается **только** для пользователей, попавших в выборку.

Для каждой выбранной задачи:
1. Отправить все ещё не отправленные события с триггерным моментом `<= now`
   (обычно одно — ближайшее). Дедуп — через существующую `notification-sent`
   (`buildReminderKey` / `buildOverdueKey`). Жёсткое окно `[scheduled, scheduled+1мин]`
   из `shouldTriggerReminderNow` больше не нужно: пропуск минуты не теряет напоминание,
   задача останется в выборке (`nextNotifyAt <= now`) и обработается в следующий прогон.
2. Отработать overdue + recurrence-перенос `dueDate` (существующая логика
   `computeNextFutureDueDate`, `updateBubbleDueDate`, `overdueSticky`,
   `overduePulseSuppressed`) — без изменений по сути.
3. **Сдвинуть `nextNotifyAt`** = `computeNextNotifyAt(обновлённый bubble, now)` и записать.
   Это снимает задачу с выборки до следующего реального события. Критический шаг — без него
   просроченная задача читалась бы каждую минуту, и экономия терялась бы.

`cleanupOldNotificationSent` (по `now.getMinutes() === 0`) остаётся: при `every 1 minutes`
минута `:00` попадается ежечасно.

### Composite index

В `firestore.indexes.json` добавить collection-group composite index:
`bubbles`: `status ASC`, `nextNotifyAt ASC`. Деплой: `firebase deploy --only firestore:indexes`.

### Backfill существующих задач

Range-фильтр по `nextNotifyAt` не вернёт документы без поля → активные задачи без backfill
выпадут из уведомлений. Разовый **локальный** скрипт на admin SDK (запустить один раз,
затем удалить): проходит `collectionGroup('bubbles').where('status','==','active')`, для
каждой пишет `computeNextNotifyAt(bubble, now)`. Запустить **до** переключения запроса в проде.

### Мелкие оптимизации (в составе A, бесплатно)

- `maxInstances = 10` на обеих функциях — потолок compute-расходов при всплесках.
- Чтение токенов только для пользователей из выборки (следствие нового запроса).

### Legacy (мёртв)

- `fetchAllUserBubbles` целиком заменяется новым запросом — его legacy-fallback уходит.
- Legacy-ветки в `updateBubbleDueDate` / `updateBubbleFields` становятся мёртвым кодом, но
  работе не мешают. Per CLAUDE.md (surgical changes) — не удаляем в рамках этой задачи,
  только отмечаем.

## Поток данных (пример)

Задача: `due = 15:00` (зона пользователя), напоминания за 60 и 10 минут, без recurrence.

| Время прогона | Выборка | Действие | Новый `nextNotifyAt` |
|---|---|---|---|
| 14:00 | попала | reminder «за 60» | 14:50 |
| 14:50 | попала | reminder «за 10» | 15:00 |
| 15:00 | попала | overdue | `null` (событий больше нет) |
| 15:01+ | не попадает | — | — |

С recurrence: на шаге overdue `dueDate` переносится на следующий цикл →
`computeNextNotifyAt` от нового `dueDate` даёт будущие моменты → задача вернётся в выборку
к нужному времени.

## Граничные случаи

- **Задача создана уже просроченной** → триггер ставит `nextNotifyAt = now` → ближайший
  прогон обрабатывает overdue/recurrence и сдвигает поле.
- **Несколько пропущенных reminder-моментов** (функция долго не работала) → шлём
  неотправленные `<= now`; дедуп не даёт дублей.
- **Часовой пояс** → `parseLocalDateTime(dueDate, tz)` уже даёт корректный абсолютный момент;
  `nextNotifyAt` всегда в UTC.
- **`overduePulseSuppressed`** → после переноса `dueDate` так же пересчитываем `nextNotifyAt`.

## Тестирование

- **Unit** (`computeNextNotifyAt`, через `exports._test`): несколько напоминаний; overdue;
  recurrence; задача в прошлом; tz; пустые/битые `notifications`.
- **Integration:** триггер проставляет поле при создании/редактировании и не зацикливается;
  scheduled-функция выбирает только задачи с `nextNotifyAt <= now`; после отправки поле
  сдвигается и задача выпадает из выборки.
- **Backfill:** на тестовом наборе все активные задачи получают корректное поле.

## Риски

- Корректность `computeNextNotifyAt` (tz, recurrence, прошедшие события) — ключевой риск,
  закрывается unit-тестами.
- Backfill обязателен до переключения, иначе старые задачи замолкнут.
- Гард от рекурсии в триггере обязателен, иначе цикл записей (расходы + лимиты).

## Решения

- `maxInstances = 10` на обеих функциях.
- Backfill — разовый локальный скрипт на admin SDK (запустить один раз, затем удалить).
