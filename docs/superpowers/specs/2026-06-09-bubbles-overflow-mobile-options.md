# Пузыри на мобиле: варианты отображения при переполнении экрана

> Статус: **брейншторм / меню вариантов** (направление ещё не выбрано). Сохранено, чтобы обдумать.
> Дата: 2026-06-09.

## Контекст и проблема

На мобильной версии при большом количестве задач пузыри не помещаются: они слипаются,
налезают друг на друга под гравитацией, мельчают и становятся нечитаемыми/непопадаемыми по тапу.

Что выяснилось в коде (определяет, какие варианты реальны):

- **Пузыри физически заперты во вьюпорте.** `createWorldBounds()` (`src/utils/physicsUtils.js`)
  ставит стены по краям экрана, а `updateWorldBounds()` там же дополнительно зажимает тела внутри
  границ (clamp по x/y, ~стр. 69–84). Рендер Matter создаётся ровно в размер вьюпорта
  (`src/hooks/useMatterEngine.js`, `Render.create` width/height = `canvasSize`). → Сейчас **все**
  задачи обязаны влезть в маленький мобильный прямоугольник.
- **Камеры / zoom / pan на канвасе пузырей нет.** Все вызовы `scale` — это ресайз отдельного тела
  (`Matter.Body.scale`), а не камера.
- **Радиус — это осознанный атрибут задачи, заданный пользователем** (`bubbleSize`, дефолт 45;
  `editBubbleSize`; хранится в Firestore как `radius`; тело создаётся `Bodies.circle(x, y, storedBubble.radius)`
  в `useMatterEngine.js`). Размер ≈ «важность». → Автоужатие пузырей размывает этот смысл.
- **Часть «решения» уже существует:** фильтры по тегам, «без тега», «только запланированные» и панель
  категорий (`src/hooks/useBubbleFilters.js`, `TasksCategoriesPanel`); в CLAUDE.md упомянут частичный
  режим списка (`Paper elevation={16}`). → Отчасти вопрос не «строить новое», а «дожать существующее».

## Варианты (от «дёшево сейчас» к «правильно и масштабируемо»)

### 1. Камера: pinch-to-zoom + панорамирование (мир больше экрана)
- **Идея:** убрать стены-по-краям-экрана, сделать мир больше вьюпорта (или «высокую банку»,
  растущую с числом задач), ввести камеру (`render.bounds` в Matter). Отдалил — видно все, свёл
  пальцы — читаешь конкретный.
- **Плюсы:** сохраняет душу приложения (физика); масштабируется практически бесконечно; самый «родной»
  мобильный жест.
- **Минусы:** самый дорогой; трогает ядро движка; нужны тач-жесты, читаемость подписей и тап-таргеты
  при отдалении; производительность на многих телах.
- **Стоимость:** высокая. **Трогает:** `physicsUtils.js` (стены/границы), `useMatterEngine.js` (рендер,
  камера), новый слой жестов.

### 2. Адаптивный размер + фокус по тапу
- **Идея:** при большом количестве глобально ужимать радиусы под площадь экрана (с **нижним порогом
  читаемости**); тап = временно увеличить пузырь.
- **Плюсы:** дёшево — масштабирование тел уже написано (`Matter.Body.scale`).
- **Минусы:** «размер = важность» размывается; при *очень* многих всё равно упрётся в пол экрана.
- **Стоимость:** низкая. **Трогает:** логику размеров в `BubblesPage.js` (где считается/применяется radius).

### 3. Показывать подмножество (опереться на уже существующее)
- **Идея, под-варианты:**
  - **(а)** дожать категории/теги — бейджи с количеством, авто-подсказка «открой категории», когда
    пузырей > N;
  - **(б)** «горизонт» — на сцене только актуальное (просрочено / скоро / важное), остальное свёрнуто
    в один пузырь «+N» или в ящик;
  - **(в)** свайп между «комнатами» по N пузырей (точки-индикатор).
- **Плюсы:** дёшево–средне; переиспользует фильтры; для таск-трекера «прятать неактуальное» — скорее
  фича (фокус).
- **Минусы:** часть задач не на виду; нужен внятный способ «показать остальные».
- **Стоимость:** низкая–средняя. **Трогает:** `useBubbleFilters.js`, `TasksCategoriesPanel`, фильтрацию
  списка пузырей в `BubblesPage.js`.

### 4. Тумблер «пузыри ↔ список/сетка» как fallback переполнения
- **Идея:** кнопка переключения в компактный скроллируемый список тех же задач (он частично есть);
  при большом количестве — предлагать/автопредлагать список.
- **Плюсы:** самый надёжный способ показать много элементов на маленьком экране; дёшево, т.к. база есть.
- **Минусы:** теряется шарм пузырей (но это запасной выход, а не замена).
- **Стоимость:** низкая. **Трогает:** существующий режим списка (`Paper elevation={16}`), переключатель в `BubblesPage.js`.

### 5. Кластеризация
- **Идея:** близкие/связанные пузыри сливаются в кластер-пузырь со счётчиком; тап «взрывает» кластер
  на его члены.
- **Плюсы:** одна сцена; масштабируется к многим; выглядит эффектно.
- **Минусы:** сложнее всех (логика группировки + анимация сборки/разлёта).
- **Стоимость:** высокая. **Трогает:** новый слой группировки поверх физики.

## Рекомендация — комбо, а не одно

- **Сейчас, дёшево:** **3а** (бейджи на категориях + авто-подсказка) и/или **4** (тумблер в список) —
  быстрый ощутимый эффект почти без риска.
- **Стратегически, чтобы реально масштабировалось и осталось «пузырями»:** **1** (камера zoom/pan).
- **Компромисс посередине:** **2** (адаптивный размер + тап-фокус), если камеру делать не хочется.

| Вариант | Стоимость | Сохраняет физику | Масштаб к «очень многим» |
|---|---|---|---|
| 1. Камера zoom/pan | Высокая | ✅ | ✅✅ |
| 2. Адаптивный размер | Низкая | ✅ | ⚠️ (упрётся) |
| 3. Подмножество/категории | Низкая–средняя | ✅ | ✅ |
| 4. Тумблер в список | Низкая | ❌ (escape hatch) | ✅✅ |
| 5. Кластеризация | Высокая | ✅ | ✅ |
