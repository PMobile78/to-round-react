## Tier 1 — утечки и стоимость (высокий приоритет)

### N1. Matter.js `Runner` не останавливается при размонтировании
`src/hooks/useMatterEngine.js:281-294`. Cleanup вызывает `Render.stop`, `World.clear`, `Engine.clear`, но **не** `Runner.stop(runner)`. Раннер продолжает тикать после размонтирования — утечка CPU/памяти, особенно при повторных монтированиях.
**Фикс:** в cleanup добавить `Runner.stop(runner)`.

### N2. `Matter.Events` слушатели не снимаются
`src/hooks/useMatterEngine.js:236, 244`. `Events.on(mouseConstraint, 'mousedown'/'mouseup', …)` регистрируются, но cleanup не делает `Events.off`. `World.clear`/`Engine.clear` их не отцепляют; замыкания держат `setBubbles`, `setSelectedBubble` и т.д.
**Фикс:** сохранить ссылки на хендлеры, в cleanup вызвать `Events.off(mouseConstraint, …)`.

### N3. Write amplification: одно изменение = чтение+перезапись всех пузырей
`src/services/firestoreService.js:66-99` (`saveBubblesToFirestore`) делает `getDocs(bubblesCol)` (чтение всех) + `batch.set(merge)` на **каждый** пузырь при любом сохранении. `updateBubbleStatus` (`:185-217`) ради смены одного поля прогоняет весь массив через эту функцию. Итог: смена статуса одной задачи = 1 чтение-всех + N записей.
**Фикс:** добавить точечный апдейт одного документа — `updateBubbleStatus` пишет только `doc(bubblesCol, id)` через `setDoc(merge)`/`updateDoc`, не трогая остальные. `saveBubblesToFirestore` оставить для массовых операций (импорт/очистка).

### N4. Cloud Function: N+1 и полностью последовательная обработка
`functions/index.js:382-451`. Планировщик раз в минуту:
- вызывает `getUserFcmTokens(userId)` **внутри `sendFcmToUser`** на каждый пузырь (`:90`) — N одинаковых чтений коллекции токенов на одного пользователя за тик;
- `await`-ит каждую отправку/дедуп-чтение/запись строго последовательно — время тика растёт линейно с числом активных пузырей, рискует упереться в таймаут.
**Фикс:** загружать токены один раз на пользователя и прокидывать в `sendFcmToUser`; параллелить независимых пользователей через `Promise.all`. Дедуп-чтения (`wasNotificationSent`) можно батчить по пользователю.

## Tier 2 — корректность / UX-баги (средний приоритет)

### N5. Отсутствует ключ перевода `categories.allCategories` в UK
Подтверждено: ключ есть в `src/locales/en/translation.json:199`, отсутствует в `src/locales/uk/translation.json`. Используется в `src/components/MobileCategorySelector.js:114,169`. Украинские пользователи видят английский фолбэк/сырой ключ.
**Фикс:** добавить ключ в `uk/translation.json`.

### N6. Отсутствуют `key` у MenuItem в MobileCategorySelector
`src/components/MobileCategorySelector.js:107` ("All categories") и `:136` (preset-строки `no-tags`, `planned-tasks`) рендерят `<MenuItem>` без `key` (только `renderCategoryOption` на `:71` ставит ключ). Предупреждения React + риск ошибок согласования.
**Фикс:** добавить стабильные `key`.

### N7. 30 fps `setInterval`, дёргающий React-состояние вхолостую
`src/pages/BubblesPage.js:1542-1562`. `setInterval(updatePositions, 33)` каждые 33 мс пересобирает массив `positions` и форсит ре-рендер оверлея текста — даже когда все пузыри в покое.
**Фикс:** перевести на `requestAnimationFrame`, привязанный к активности движка, либо пропускать `setPositions`, если позиции не изменились (сравнение с предыдущими).

### N8. Тема пересчитывает все пузыри при любом изменении `bubbles`
`src/pages/BubblesPage.js:446-476`. Эффект с deps `[themeMode, bubbles, tags]` гоняет `bubbles.forEach` перекраски, хотя по сути должен реагировать только на `themeMode`/`tags`. Любой drag/создание/правка пузыря зря перезапускает цикл.
**Фикс:** убрать `bubbles` из зависимостей (перекраска цвета не зависит от позиций) либо разнести на отдельный эффект.

## Tier 3 — мелочи и гигиена (низкий приоритет, опционально)

- **N9.** `HtmlRenderer` (`src/components/HtmlRenderer.js`) санитизирует HTML в `render` и не обёрнут в `React.memo` — в списке задач каждый ре-рендер родителя пересанитизирует весь HTML. Фикс: `React.memo` + `useMemo` на результат `DOMPurify.sanitize`.
- **N10.** `firebaseMessaging.js:77-84` пишет `createdAt: serverTimestamp()` при каждом merge токена — затирает исходное время создания. Фикс: писать `createdAt` только при создании.
- **N11.** `saveTagsToFirestore` (`firestoreService.js:285-289`) — `setDoc` без `{merge:true}`, перезаписывает весь документ тегов. Низкий риск, но несимметрично с пузырями.
- **N12.** `cleanupOldNotificationSent` (`functions/index.js:245-254`) — один `batch` без учёта лимита 500; при накоплении >500 старых записей `commit()` упадёт. Фикс: чанковать удаление.

## Сознательно НЕ включено (проверено — не баги / слишком рискованно)

- **Таймзона Cloud Function** (`parseLocalDateTime`/`formatLocalDateTime`, server-local): осознанное проектное решение — приложение хранит время без TZ, функция трактует его в своей зоне. Менять рискованно без понимания, где живут пользователи; требует отдельного обсуждения.
- **`useEditorResize` порядок объявления** — работает за счёт замыканий, не дефект.
- **`AddNotification` `onDelete(idx)` по sorted-индексу** — оба пути добавления (`handleSave`, `handleCustomSave`) сортируют массив перед `onAdd`, так что родительский массив хранится уже отсортированным и индексы совпадают. `key={idx}` оставить как мелочь, не критично.
